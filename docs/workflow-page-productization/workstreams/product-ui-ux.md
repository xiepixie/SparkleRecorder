Owner 2 负责把 Workflow 页面做成用户能完成任务的产品界面。当前页面不能按完成验收；下一阶段先用 fixture 重建可用交互，再接 live projection。

视觉和交互验收标准以 [../06-workflow-visual-ux-contract.md](../06-workflow-visual-ux-contract.md) 为准。Owner 2 的 UI 代码如果偏离该合同，必须先在 [../02-backend-frontend-contract.md](../02-backend-frontend-contract.md) 或本文件记录例外原因。

## Current Product Mandate

Workflow 页面要按 Pro 级 macOS 原生生产力工具来设计。它不是 Web dashboard，也不是由一组大色块按钮驱动的配置页。UI 在静止态应该退到背景里，让用户的宏、依赖、时间轴、条件和运行证据成为主视觉。

Owner 2 的下一轮 UI 重做必须围绕四个体验目标：

- 克制视觉：移除普通控件上的高饱和全宽底色和无必要 `.controlSurface`。
- 所见即所得：Macro Library 拖到 graph/list 是创建 task 的主路径。
- 直观看懂依赖：中心区域直接表达 success、failure、timeout、condition then/else，而不是只靠右侧 Inspector。
- 看它自己跑：运行中节点、触发边、资源等待和失败证据在 graph + timeline 中同时可见。

当前页面如果仍以“点击按钮 -> 右侧 Inspector 填依赖”为主要工作方式，只能视为过渡实现。最终产品路径必须像 Pro app 的素材编排：Macro Library 是 source bin，中心 FlowGraph/list 是编排台，Inspector 只是精修面板。

Owner 2 的视觉审查顺序固定为：先看用户数据是否成为主视觉，再看状态反馈是否清楚，最后才看普通命令是否方便。不能用更响亮的 `.controlSurface` 或全宽高饱和按钮来弥补交互不清；交互不清时应补 connector handle、插入线、hover/focus outline、菜单、上下移动按钮和明确的 drop target。

当前产品定位的直接含义：Workflow 页面要像 Pro macOS 编排台，而不是 Web 表单。普通按钮和工具行默认应“隐身”，让 graph、timeline、condition branch 和 run evidence 成为主视觉。Macro Library 是 source bin；用户把 `SavedMacro` 拖到画布/list 后才生成 task instance。运行时反馈要通过细描边、hairline progress、状态 chip 和资源时间轴让页面活起来，不能用整块高亮背景或 Bootstrap 式全宽按钮制造存在感。

### Current Non-Negotiable Product Gates

Owner 2 后续 UI 切片先按这些门槛自检，再谈 polish：

| Gate | Required Shape | Reject When |
| --- | --- | --- |
| Pro native restraint | 普通控件低对比、少填充，用户数据和运行状态是主视觉 | 页面第一眼像 Web dashboard、Bootstrap 按钮组或彩色控制面板 |
| Source-bin authoring | Macro Library 拖入 graph/list 是主路径，点击添加是辅助路径 | 用户必须先点按钮或打开 Inspector 才能创建有效 workflow |
| Direct dependency graph | 节点连接点、语义箭头、trigger label、then/else/timeout 分支在中心区域可读 | 条件和依赖只在右侧表格里能看懂 |
| Runtime feedback | running/waiting/failed/timeout 同步出现在 graph、timeline、inspector | 运行时画布像静态页面，只有右侧字段变化 |
| Stable WYSIWYG | 拖入、移动、插入线、临时连线和真实 mutation 完全一致 | 插入提示和实际位置不一致，或新 task 需要刷新/重新定位才出现 |

## Current Product Direction

下一轮 Owner 2 的目标不是继续给现有页面补按钮，而是把 Workflow 从“配置面板”推进成“可视化编排台”。验收时先看用户是否能凭直觉完成动作，再看 Inspector 字段是否齐全。

必须同时满足：

- Macro Library 看起来像素材库/source bin，不像一组大色块添加按钮；拖出的 `SavedMacro` 只派生 `AutomationTask`，不污染宏模板。
- 中心 FlowGraph/list 是主操作区：拖入、移动、插入、连线、删除和选择都应在这里可见、可预测。
- 依赖和条件不再只靠右侧表格解释；success/failure/timeout/condition then/else、join policy 和等待原因要能在中心区域或 timeline 直接读到。
- 运行时画布必须“活起来”：当前 task、触发边、foreground input 等待、OCR/视觉条件等待、失败证据入口需要同步出现在 graph/timeline/inspector。
- 普通控件不靠 `.controlSurface`、高饱和全宽底色或整块填充来获得注意力；需要强调时优先使用 connector handle、insertion line、selection outline、status hairline、menu 和原生按钮。

Owner 2 不能把“点击按钮 -> 右侧 Inspector 配依赖”作为最终主路径。它可以保留为辅助和可访问性路径，但主体验必须是所见即所得拖拽编排。

### Repeat-Until Product Direction

视觉 Repeat-Until 的产品形态采用 graph-first structured loop node，详细设计维护在 [../../semantic-recording-ai/16-visual-repeat-until-design.md](../../semantic-recording-ai/16-visual-repeat-until-design.md)。

Owner 2 不应把 Repeat-Until 做成任意 dependency cycle，也不应只把它藏成 Inspector 里的线性缩进配置。画布上应有一个结构化 loop container：用户能看到 `Do`、`Until`、return ribbon、max attempts、timeout、polling、failure policy 和 source/runtime evidence。底层 runtime 必须等待 Owner 1 提供结构化 loop state / attempt evidence contract；在此之前，UI 可以在 Draft Preview 里解释 loop intent，但不能把 graph back-edge 当作已支持运行语义。

Macro Editor 的首版职责是提供“把选中行为变成 Repeat Until...”的入口和 Review/Draft Preview handoff；复杂循环编辑属于 Workflow surface。OCR waits 继续表达文字；图标、按钮、区域变化和像素状态应进入 visual condition detector，而不是塞进 OCR。

## Hard Boundary

- Do not call Player, scheduler, repository, arbiter, or condition evaluator directly.
- Do not implement scheduler, dependency, retry, timeout, or resource queue semantics in SwiftUI.
- Do not run graph searches or file IO inside SwiftUI `body`.
- Do not depend on drag-only interaction; every critical edit needs a button/menu/inspector path.
- Do not treat a backend capability as real until Owner 1 marks it `ready-for-ui`.
- Do not use saturated full-width control backgrounds for ordinary Workflow controls.
- Do not wrap normal nodes, edge rows, or toolbar actions in `.controlSurface` just to make them louder; reserve emphasis for selected, dangerous, or live-running states.
- Do not make Inspector the only creation path; drag from Macro Library and direct graph/list authoring are first-class flows.
- Do not show dependency or condition semantics only after selection; graph/list must communicate trigger meaning inline.

## P0 Interaction Contract

这些不是 polish，而是 Workflow 页面是否可用的最低线。

| User Action | Required UI Behavior | Failure Mode |
| --- | --- | --- |
| Drag `SavedMacro` from Macro Library to graph | Create a task instance at the drop position and show it immediately | Task only appears after refresh, reposition, or inspector edit |
| Drag `SavedMacro` to list | Show insertion before/after and commit to the same position | Indicator says after, actual insert before |
| Move an existing task | Adjacent insertion is exact even when timestamps match or a condition spans time | Move down skips over a long-running/same-time task |
| Drag connector from task to task | Show temporary line, valid targets, trigger choice, then create dependency | Dependency only possible from right panel |

Every P0 interaction needs a non-drag path: move up/down, insert before/after, connect selected task, delete edge, and open inspector details.

## Immediate UX Direction

Owner 2 should treat the current Workflow page as a first-pass shell, not as the final interaction model. The next UI slice should prioritize these user-visible corrections:

1. Make Macro Library drag-to-canvas/list the obvious creation path.
2. Make graph/list dependency authoring direct: drag a connector, see an arrow, choose or edit trigger inline.
3. Keep Inspector as refinement, not as the only place dependencies are configured.
4. Replace Web-like full-width highlighted controls with quiet native macOS controls.
5. Add runtime feedback states to fixture UI before live wiring: running, waiting upstream, waiting resource, condition waiting, completed, failed, timeout.
6. Ensure every drag operation has an exact non-drag alternative: move up/down, insert before/after, connect selected to selected, delete edge.

Style changes should be evaluated by what the user sees at rest. If the page at rest looks like a grid of big colored web buttons, it is not acceptable for this product tier. At rest, the page should look like a professional editing surface; emphasis appears only while interacting or while a run state requires it.

The first review should use three screenshots: idle workflow, drag/link interaction, and running workflow. Passing idle alone is not enough; passing drag and running states with loud persistent backgrounds is also not enough. The same restrained language must hold across all three states.

## Visual Design Rules

- Treat Workflow as a Pro macOS production surface, not a Web dashboard.
- Ordinary nodes and toolbar controls should be quiet: light border, subtle material, minimal fill.
- Use color only as semantic accent for success/failure/timeout/running/waiting, not as large decoration.
- Selected and running states may use a restrained outline, hairline progress, or breathing stroke.
- Avoid full-width saturated backgrounds and Bootstrap-like button blocks.
- Avoid `.controlSurface` on ordinary full-width actions; use it only when a native platform control truly needs that treatment.
- Do not make ordinary action rows look like primary marketing CTA buttons; use icon buttons, menus, handles, insertion lines, and native selection states instead.
- Macro Library drag targets, dependency handles, and insertion indicators should be visible during interaction but quiet at rest.
- Runtime feedback should make the graph feel alive through progress/outline/status markers, not through large animated backgrounds.
- Runtime animation must not resize cards, shift layout, or distract from the graph.
- Replacing strong backgrounds must preserve affordance: use connector handles, hover outlines, insertion lines, menus, focus rings, and selection outlines so users still know where to act.
- Macro Library should feel like a source bin/material library in a Pro app: quiet at rest, clear during drag, never a panel of loud web cards.

## Runtime Feedback Rules

- Current task: subtle animated outline, top progress hairline, or compact activity indicator.
- Waiting resource: neutral status marker with user wording such as “等待鼠标键盘空闲”.
- Waiting condition: read `conditionProgress` and show the watched condition, target/detail copy, polling cadence, and elapsed/remaining timeout without expanding the node.
- Completed: low-emphasis success marker; do not permanently turn the whole node green.
- Failed/timeout: small semantic badge plus evidence affordance; details open on demand.
- Triggered dependency: briefly emphasize the edge or label, then return to low contrast.
