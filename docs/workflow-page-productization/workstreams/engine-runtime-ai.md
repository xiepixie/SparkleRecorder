## Hard Boundary

- Do not import SwiftUI in core/runtime logic.
- Do not call live Player/OCR/file IO from reducer tests.
- Do not let CLI write internal state directly; import must compile draft into accepted workflow values and persist through repository/reducer boundaries.
- Do not build MCP in this phase.
- Do not ask UI to infer scheduler, dependency, retry, timeout, or resource queue semantics.

## Ready-For-UI Notice Queue

Ready-for-UI first pass:

- `workflow draft validate draft.json --json`: returns `sparkle.cli.result.v1`, validation payload, warnings, errors, nextActions.
- `workflow macros --json`: returns `sparkle.cli.result.v1` macro catalog data with exact macro IDs, search filtering, lightweight stats, resource requirement, notes, and surface summaries. It is read-only and does not expose recorded event payloads.
- `workflow draft simulate draft.json --json`: returns `sparkle.cli.result.v1` deterministic draft simulation data with planned steps, branch decisions, resource timeline, skipped task keys, validation warnings, and scenario support such as `timeout:<taskKey>`.
- `workflow draft init/inspect/normalize/task add/task set/task remove/schedule set/condition set/dependency add/dependency set/dependency remove`: returns `sparkle.cli.result.v1` edit data with the edited external draft document, current validation state, changed task/dependency keys, and optional `wrotePath`. These commands only write when `--out` is explicit and never touch `automations.json`.
- `workflow draft patch draft.json patch.json --json`: returns `sparkle.cli.result.v1` edit data after sequentially applying `sparkle.workflow.patch.v1` operations. First pass supports `addTask`, `setTask`, `removeTask`, `setSchedule`, `setCondition`, `addDependency`, `setDependency`, `removeDependency`, and `normalize`. It only writes when `--out` is explicit and never touches `automations.json`.
- `workflow list/show/export`: returns `sparkle.cli.result.v1` read/export data for existing repository workflows. List/show read workflow and run-history summaries; export returns a `sparkle.workflow.draft.v1` document with task/dependency key maps and `lossyWorkflowExport` warnings when draft v1 cannot represent internal details such as concrete OCR region bounds. These commands do not run Player or mutate app repository state.
- `workflow status [workflow-id] --json`: returns `sparkle.cli.result.v1` read-only workflow/task runtime status summaries from repository workflows plus run history. It reports user-facing labels/details for idle/planned/waiting/running/completed/needs-attention states, including “等待鼠标键盘空闲” for resource waits. It does not create runs, cancel runs, acquire resources, start Player, or mutate repository state.
- `nextScheduledOccurrence` projection: `AutomationViewProjection` computes workflow-level and task-level next scheduled occurrence values from once/repeating schedules and run history, so UI does not parse schedule semantics in SwiftUI.
- Repeating scheduled run generation first pass: reducer `clockTick` now asks `AutomationSchedule.nextDueOccurrence(onOrBefore:excludingScheduledStartTimes:)` for the earliest due unrepresented occurrence and creates a fresh `AutomationTaskRun` for that scheduled time. This is runtime-core behavior, not a UI contract to compute schedules.
- Resource waiting queue/resume first pass: `.resourceLeaseDenied` now keeps the run in `waitingForResource` without terminal outcome, `clockTick` retries existing waiting runs, and terminal release/panic release refreshes the waiting queue before newly-created downstream work can jump ahead.
- Resource waiting reason projection first pass: `AutomationResourceWaitingProjection` now appears on task nodes and resource timeline items for runs in `waitingForResource`. UI can show detail text, resources, priority, waitingSince/waitedDuration, and active lease blockers without reading reducer internals or reimplementing resource queue logic in SwiftUI.
- Timeout watchdog first pass: `.clockTick` now expires queued/running runs whose `AutomationTask.timeout` deadline has passed. Macro tasks emit `.cancelPlayer` before terminal cleanup; `completeRun` handles lease release, `.persistRun`, waiting-resource retry, and timeout downstream resolution. Resource waiting time is deliberately not counted as task runtime timeout.
- Retry policy first pass: terminal failure/timeout outcomes now create another `AutomationTaskRun` in the same execution when `AutomationRetryPolicy.maxAttempts` allows it. Retry runs increment `attempt`, preserve upstream run IDs, honor fixed/exponential backoff through `earliestStartTime`, and suppress failure/timeout downstream branches until the final attempt.
- Runtime feedback projection first pass: `AutomationViewProjection` now exposes `timeoutCountdown` and `retryAttemptSummary` on both `AutomationTaskNodeProjection` and `AutomationResourceTimelineItem`, so UI can render timeout progress and retry attempt state without recomputing runtime semantics in SwiftUI.
- Join policy first pass: `AutomationTask.joinPolicy` now controls dependency resolution for multi-input nodes. `all` preserves the old behavior, `any` selects the earliest ready incoming branch, and `firstMatched` locks the first completed incoming branch for that target run. Draft JSON supports task-level `joinPolicy`, and `workflow draft task add/set --join-policy` plus patch/import/export preserve it.
- Visual condition core spec first pass: `AutomationVisualCondition` now carries region/image/baseline/pixel/color/threshold intent for `regionChanged`, `imageAppeared`, `imageDisappeared`, and `pixelMatched`. Draft validator/import/export/editor/patch/CLI preserve those fields, condition evaluator clients can route visual evaluation through an injected provider, and the default live factory now uses the app-edge visual evaluator.
- Visual condition progress projection first pass: `AutomationConditionProgressProjection` now appears on task nodes and resource timeline items for condition tasks. UI can show what is being watched, whether it is actively polling, its polling interval, and condition timeout progress without unpacking condition enums inside SwiftUI.
- Live visual condition evaluator first pass: `AutomationVisualConditionEvaluatorClient.live` captures the current display with `ScreenCaptureService`, evaluates `pixelMatched` against live pixels, supports `imageAppeared` / `imageDisappeared` via injected template images, and supports `regionChanged` via injected baseline images. Core remains free of `CGImage` / ScreenCapture dependencies; `AutomationVisualCondition.searchRegionResolution(in:)` keeps visual region spaces aligned with OCR.
- Visual asset/region registry for AI drafts first pass: `sparkle.workflow.draft.v1` now has optional `visualAssets.regions/images/baselines`. Import resolves `regionRef` through that registry for OCR and visual conditions, export writes concrete OCR/visual regions into `visualAssets.regions`, and validation warns on missing region/image/baseline refs when a registry is present.
- Visual assets persistence and workflow package provider binding first pass: `AutomationWorkflow.visualAssets` preserves package regions/images/baselines across internal workflow/package round trips, and app target `AutomationVisualAssetImageProviders.workflowPackages(...)` builds request-aware image/baseline providers from package-local `visualAssets` paths. Package-root retention now has a first pass through the visual asset roots manifest and runtime host restoration path; external file copy/storage remains open.
- Visual asset package-root retention first pass: confirmed CLI imports, AI Draft Preview imports, and workflow package imports can record or clean local package roots in `automation-visual-asset-roots.json`. `LiveAutomationRuntimeHost` can reload that manifest with repository workflows and rebuild workflow-scoped image/baseline providers after app restart/reopen. This does not copy external assets or solve moved/deleted files.
- Runtime App-host handoff mailbox/status first pass: `AutomationRuntimeHandoffCommand` carries manual-start and cancel-run requests through `automation-runtime-handoff.json`. CLI `workflow run/cancel --handoff app` enqueues commands and returns `AutomationRuntimeHandoffPayload`; `LiveAutomationRuntimeHost` polls the mailbox, dispatches commands through the existing runtime session, and writes `AutomationRuntimeHandoffReceipt`; CLI `workflow handoff status <command-id>` returns `pending`, `dispatched`, `failed`, or `missing`, plus repository-backed `runs` snapshots and optional `workflowStatus` when the receipt identifies run IDs. This does not wake a non-running App or push a live progress stream back to the CLI.
- `workflow import draft.json --dry-run --json`: returns `sparkle.cli.result.v1` import preview data with compiled internal `AutomationWorkflow`, stable task/dependency ID maps, macro resolutions, import-blocking errors, and warnings such as unresolved OCR region references. It is read-only and does not write the repository.
- `workflow import draft.json --confirm --json`: returns `sparkle.cli.result.v1` confirmed import data after compiling the same draft, routing the write through reducer `.upsertWorkflow` / `.persistWorkflows`, and saving via repository. It does not run Player or mutate `SavedMacro`.

Next candidates:

- Daemon/background runtime handoff for waking a non-running App, long-running sessions, and push-style live progress/result streaming beyond polling repository-backed handoff status.
- Broader managed template/baseline storage or migration policy, plus visual-condition product evidence.
